# Narman — Hybrid Lovable Cloud + Supabase Pribadi

Panduan setup setelah clone repo dari GitHub ke project Supabase pribadi
(misal untuk hosting di Cloudflare Pages/Workers), TANPA membawa key
internal Lovable Cloud.

## 1. Prinsip Hybrid

- **Saat develop di Lovable**: project memakai Lovable Cloud (Supabase
  internal). Key-nya disuntik otomatis oleh Lovable dan **TIDAK** ada di
  repo GitHub (lihat `.gitignore` — `.env` di-exclude).
- **Saat deploy ke Cloudflare** (atau host lain): isi sendiri kredensial
  Supabase pribadi sebagai **environment variables** di dashboard
  Cloudflare. Repo hanya berisi `.env.example` sebagai referensi.

## 2. File di folder ini

| File | Fungsi |
|------|--------|
| `01.schema.sql`        | Semua tabel, enum, fungsi, RLS, trigger di schema `public`. |
| `02.data-publik.sql`   | Data master publik: OPD, desa, kategori & layanan, pejabat, berita, data terpadu, app_setting. |
| `03.data-usee.sql`     | Data user lama (profil, role, permohonan, dll). Lewati untuk DB kosong. |
| `04.storage-dan-auth.sql` | Bucket storage + policies + trigger `on_auth_user_created`. |
| `05.pg-cron.sql`       | Jadwal cron harian (backup snapshot & storage cleanup). |

Semua file ditulis agar bisa **dijalankan langsung di Supabase SQL Editor**
project pribadi.

## 3. Urutan Setup di Supabase Pribadi

1. Buat project baru di [supabase.com](https://supabase.com).
2. Buka **SQL Editor** → jalankan berurutan:
   ```
   01.schema.sql
   02.data-publik.sql
   04.storage-dan-auth.sql
   ```
   (Lewati `03.data-usee.sql` jika project baru tanpa user lama.)
3. **Authentication → Providers**: aktifkan **Email**, dan **Google**
   bila perlu (isi Client ID / Secret Google sendiri).
4. **Authentication → URL Configuration**: tambahkan domain Cloudflare
   Anda ke "Site URL" dan "Redirect URLs".
5. **Database → Extensions**: aktifkan `pg_cron` dan `pg_net`.
6. Edit `05.pg-cron.sql`, ganti `<PROJECT_URL>` & `<CRON_SECRET>`, lalu
   jalankan.
7. Buat super admin pertama:
   - Daftar lewat halaman `/auth` di web yang sudah dideploy.
   - Di SQL Editor:
     ```sql
     UPDATE public.user_roles
       SET role = 'super_admin'
       WHERE user_id = (SELECT id FROM auth.users WHERE email='you@example.com');

     UPDATE public.profiles
       SET verified_at = now()
       WHERE id = (SELECT id FROM auth.users WHERE email='you@example.com');
     ```

## 4. Environment Variables di Cloudflare

Set di **Cloudflare Pages/Workers → Settings → Environment Variables**
(jangan commit ke repo):

| Nama | Sumber | Scope |
|------|--------|-------|
| `VITE_SUPABASE_URL`             | Supabase → Project Settings → API → URL | Build & Runtime |
| `VITE_SUPABASE_PUBLISHABLE_KEY` | API → `anon` / `publishable` key         | Build & Runtime |
| `VITE_SUPABASE_PROJECT_ID`      | Bagian subdomain URL (`xxx`.supabase.co) | Build & Runtime |
| `SUPABASE_URL`                  | sama dengan `VITE_SUPABASE_URL`          | Runtime         |
| `SUPABASE_PUBLISHABLE_KEY`      | sama dengan `VITE_SUPABASE_PUBLISHABLE_KEY` | Runtime      |
| `SUPABASE_SERVICE_ROLE_KEY`     | API → `service_role` key (RAHASIA)       | Runtime         |
| `CRON_SECRET`                   | string acak, samakan dgn yang di `05.pg-cron.sql` | Runtime |
| `LOVABLE_API_KEY` *(opsional)*  | hanya jika tetap pakai AI Gateway Lovable | Runtime        |

Tinjau juga `.env.example` di root repo.

## 5. Catatan Keamanan

- **JANGAN PERNAH** commit `.env` ke GitHub. File `.gitignore` sudah
  meng-exclude `.env*` kecuali `.env.example`.
- `SUPABASE_SERVICE_ROLE_KEY` HANYA dipakai oleh kode server
  (`createServerFn`, server routes). Tidak boleh prefix `VITE_`.
- Setelah ganti key (rotate), redeploy Cloudflare untuk pick up env baru.
